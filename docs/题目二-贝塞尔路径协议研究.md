# 题目二 · 贝塞尔路径协议研究（精细化绘制，突破图元词汇）

> 分支 `research/fine-detail-engine`。目标：让作画不再局限于 10 个几何图元的拼装，
> 达到绘本/贴纸级的精细插画质量，同时守住项目底线（可编辑场景图、语音可改、不走不可编辑栅格生图）。

## 1. 问题：图元拼装的画质天花板

现状管线 = LLM 把图案拆成 ~26 个几何图元（circle/rect/triangle/line/path…）→ 场景图 → 渲染。
天花板就在"**LLM 拼图元**"：无论渲染器是 Konva 还是自由画笔，画出的都是同一坨方块感构图
（实测"画一只羊"= 4 条相同矩形腿 + 3 个圆堆的羊毛）。换渲染器不改画质——画质杠杆在**协议表达力**。

## 2. 实验验证（2026-06-13，doubao-seed-2.0-pro）

给 LLM 换一套协议：只输出 `{viewBox, paths:[{d, fill, stroke, strokeWidth}]}`，
用 SVG path 的 `d`（三次贝塞尔 C/Q）画对象，多 path 从后到前。渲染为 SVG 静态查看。

- **实验 A**「侧面卡通绵羊」：17 条贝塞尔路径，**整体轮廓极佳**（蓬松云朵羊毛身体、棕腿带蹄），
  但**五官差**——侧面单眼挤一边、器官过大、布局乱。
- **实验 B**「可爱卡通绵羊」+ prompt 强调"五官精致/对称/比例协调 + 正面视角 + 绘本插画水准"：
  19 条路径，**质变**——对称大眼带高光、小鼻、微笑嘴、**粉色腮红**、毛耳带浅内耳、卷额毛、棕蹄，
  **贴纸级可爱卡通羊**。

**结论**：
1. 贝塞尔路径协议**彻底成立**：整体轮廓 ✓ + 精细五官 ✓，远超图元拼装，直接突破图元词汇天花板。
2. 五官质量**可由 prompt 拉满，且不靠教解剖**——只给"质量预期（对称/协调/可爱/插画级）+ 视角"，
   造型仍出自 LLM 自身认知（符合"别在 prompt 里教动物长什么样"的原则）。
3. **正面/对称视角**对五官摆位帮助极大；终态截图清晰矢量观感最佳（非手绘）。

## 3. 协议设计（命名多 path 插进现有场景图）

用户决策：**每条 path 作为一个带名字的场景对象插入现有场景图**，复用现有语音编辑/移动/改色/选中，
保留可编辑性。

### 3.1 DSL / Schema
- create 新增字段 `d?: string`（SVG path data）+ 形状 `shape: 'vpath'`（vector path）。
- 复用 `name`（部件名，语音编辑按名作用）、`fill`/`stroke`/`strokeWidth`/`gradient`/`opacity`/`shadow`、`z`。
- 坐标：`d` 落在统一 viewBox（建议直接用画布系 1024×768，或 512 再整体缩放放置）。
  整幅插画 = 一组共享初始位置/缩放的命名 vpath；独立编辑某部件 = 改该对象自身 transform。

### 3.2 渲染（建议：清晰矢量优先）
- vpath 用 **Canvas Path2D**（`new Path2D(d)` → fill + stroke）**清晰渲染**——贴纸级观感（实验 B 的效果）。
- 决策点：路径插画**走清晰矢量、非手绘墨线**（质量优先；自由画笔的手绘风对精细插画是减分）。
  → 这意味着主渲染需支持 vpath 的 crisp 绘制（FreehandSceneStage 混合渲染：vpath crisp、图元仍可手绘；
  或为 vpath 走独立清晰矢量层）。**需与"已合入的自由画笔主渲染器"对齐**。
- 可选：逐 path 渐进上屏（描边 → 填充）做"作画过程"动画。

### 3.3 bbox / 可编辑性
- vpath 的包围盒需从 `d` 计算（解析或沿路径采样取 min/max），供选中高亮、相对定位、resize。
- 语音"把眼睛变大/换个颜色/往左移" → 按 path `name` 命中该 vpath 对象，作用其 transform/样式（多轮复用现有 §5.x）。
- 难点：path 间相对关系（移头不连身）比图元弱——首版接受"独立部件编辑"，相对约束后续再补。

### 3.4 Prompt：插画模式
- 在 plan 之上新增"细节插画"出图：要求多**命名** path、从后到前、正面/四分之三、
  质量预期（对称五官、腮红高光、配色协调、占画布 ~70%）、坐标在画布内。**不教具体解剖坐标**。
- 校验：每条 path 的 `d` 合法（M 起手、含曲线）、name 非空。

### 3.5 流式
- 逐 path 增量透传 + 上屏（复用 v1.4 OpStreamExtractor 思路），"边画边出"。

## 4. 与现有引擎关系
- vpath 与 10 个图元**并存**：简单形状（背景矩形、纯色块）仍可用图元；主体/有机物用 vpath。
- 自由画笔渲染器：对 vpath 默认走清晰矢量（手绘风可作为可选呈现开关）。

## 5. 风险 / 开放问题
- vpath bbox 计算精度（影响选中/相对定位/resize）。
- LLM 偶发坐标越界 / `d` 语法错误 → 需校验 + 降级。
- 编辑时 path 间关联（移动主体带动附件）——首版弱。
- 性能：单图几十条 path 的 Path2D 渲染（可接受）；流式逐条。
- 与自由画笔主渲染器的渲染分工（crisp vs 手绘）需定稿。

## 6. 分阶段计划
- **P1（出图，只读）**：schema 加 `vpath`+`d`；interpreter 建 vpath；渲染 crisp Path2D；
  插画 prompt 模式；端到端"画一只羊"出清晰插画。先不做编辑。
- **P2（编辑）**：vpath bbox；命名 path 的语音编辑（变大/改色/移动）、选中高亮、多轮。
- **P3（打磨）**：逐 path 流式渐进绘制；越界/语法校验与降级；图元↔vpath 混合构图调优。

## 7. 实验复现
脚本（本地、不入库）：读 backend/.env 的 ARK 密钥，POST `…/chat/completions`，
system 给"只输出 {viewBox,paths} + 质量预期"，user 给"画：<对象>"。
渲染：把 paths 拼进 `<svg viewBox>` 静态页查看。两次实验产物见提交说明。
