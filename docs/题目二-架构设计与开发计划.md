# 题目二：AI 语音绘图工具 — 架构设计与开发计划

> 一款纯语音控制的绘图工具：用户不使用鼠标键盘，仅靠说话完成绘图创作。

---

## 1. 核心设计目标与挑战

| 挑战 | 设计应对 |
|------|---------|
| 指令理解的准确性与容错性 | 三层指令理解架构（规则快路径 → LLM 结构化解析 → Agent 拆解），同音词纠错，低置信度语音确认 |
| 语音到绘图操作的响应延迟 | 流式 ASR + 本地 VAD 切分 + 部分转写投机解析；简单指令目标 < 500ms，复杂指令 < 3s |
| 复杂指令的拆解与执行 | 统一绘图 DSL 作为中间表示，LLM 把"画一个雪人"编译为多条原子操作序列，逐条执行可中断 |
| 纯语音交互（无降级输入） | 常开麦克风 + VAD 自动断句；确认/取消也走语音；TTS 播报反馈形成闭环 |

### 2.1 关键决策

**决策一：统一绘图 DSL 作为系统契约。**
所有理解层（规则 / LLM / Agent）的输出都是同一套 JSON 原子操作（Op），执行引擎只认 DSL。
好处：① undo/redo 天然成立（命令模式）；② 可以用文本直接灌 DSL 做自动化测试，不依赖语音；③ LLM 输出可校验（JSON Schema 验证失败即重试/降级）。

```jsonc
// DSL 示例（规范定义见 交互协议规范 §1.3-1.4）
{ "op": "create", "shape": "circle", "name": "太阳", "fill": "#FFD700",
  "at": { "ref": "canvas", "anchor": "top-right", "offset": [-80, 80] }, "size": 50 }
{ "op": "move",   "target": { "byName": "太阳" }, "delta": [0, 40] }
{ "op": "style",  "target": { "byFocus": true }, "fill": "red" }   // byFocus = 指代消解："它/刚才那个"
{ "op": "undo" }
{ "op": "group",  "targets": [{ "byId": "circle#1" }, { "byId": "circle#2" }], "name": "雪人" }
```

**决策二：三层指令理解，按复杂度路由（也是成本与延迟的核心控制点）。**

| 层 | 处理什么 | 延迟 | 成本 |
|----|---------|------|------|
| ① 规则快路径（本地） | 高频简单指令：撤销/重做/清空/放大/换颜色/"画一个红色的圆" 等带槽位的模板句式 | < 50ms | 0 |
| ② LLM 单轮结构化解析 | 带空间关系、相对修饰的指令："在房子左边画一棵比它矮的树" → 输出 DSL Op 数组 | 1~2s | 一次调用 |
| ③ Agent 多步拆解 | 创作型指令："画一个雪人""画一幅日落海景" → 规划为多条 Op，逐条执行并播报进度 | 2~5s | 1~2 次调用 |

路由器先用规则层尝试匹配（覆盖目标 70%+ 的日常指令）；不命中再升级到 LLM，prompt 中携带**场景图摘要**（画布上有哪些对象、名称、位置、最近操作），使 LLM 能消解"它/那个圆/刚才画的"等指代。

**决策三：延迟优化靠流式与投机执行。**
- ASR 用 WebSocket 流式接口，拿 partial 转写就开始做规则层预匹配（投机解析），final 到达时若与 partial 一致直接执行，体感延迟≈说完即画。
- VAD 在本地（Silero VAD WASM）做断句，静音 ~500ms 即切段送 ASR，不等用户手动停止（也没有按钮可按）。
- TTS 反馈短句化（"好的""已撤销"），长反馈只在出错/歧义时使用。

**决策四：容错设计。**
- **同音词纠错**：ASR 结果先过领域词表纠错（"园→圆"、"花→画"、"原型→圆形"），用拼音相似度匹配。
- **低置信度确认**：删除/清空等破坏性操作，或解析置信度低时，TTS 询问"你是想清空整个画布吗？"，语音"确认/取消"决定执行；其余操作直接执行 + 依赖 undo 兜底（宁可做错可撤销，不要频繁打断确认）。
- **指代消解**：维护"焦点对象"（最近创建/操作的对象），"把它变大""再画一个"都能解析。
- **失败兜底**：LLM 输出 JSON 校验失败自动重试一次；仍失败则 TTS 播报"没听懂，请换个说法"。

## 3. 技术选型

| 模块 | 选型 | 理由 |
|------|------|------|
| 前端框架 | React + TypeScript + Vite | 生态成熟，快速搭建 |
| 画布 | Konva.js (react-konva) | 自带场景图/对象模型/序列化，契合 DSL 执行引擎 |
| VAD | @ricky0123/vad-web (Silero VAD, WASM) | 本地运行零成本，断句准 |
| ASR | 火山引擎豆包流式语音识别（双向流式优化版 bigmodel_async，经 backend 网关转发）；Web Speech API 兜底 | 大模型 ASR 准确率高、partial 仅在结果变化时返回（天然适配字幕）；无密钥自动降级 mock 上游 |
| LLM | 七牛云大模型推理服务（如 deepseek-v3），JSON Schema 约束输出 | 函数调用式结构化输出，便于校验 |
| TTS | 七牛云 TTS；浏览器 speechSynthesis 作兜底 | 兜底方案零成本零依赖 |
| 后端 | 轻量 Node 代理（仅做密钥隐藏与 ASR/LLM 转发） | 密钥不下发前端；无业务状态，保持薄 |

## 4. 指令能力清单（计划支持）

按优先级分级，P0 必做、P1 应做、P2 有余力做：

- **P0 画布与系统**：清空、撤销、重做、（语音激活开关）
- **P0 基础图形创建**：圆/矩形/三角形/直线/文字 + 颜色/大小/位置槽位（"在左上角画一个大的蓝色矩形"）
- **P0 对象操作**：移动（方向+距离）、缩放、改颜色、删除；支持"它/刚才那个"指代
- **P1 空间关系**：相对定位（"在房子右边""比它高一点"）、对齐
- **P1 复杂创作拆解**：组合图案（雪人/房子/笑脸/太阳）由 Agent 拆解为原子 Op 序列执行
- **P1 选择与命名**：给对象命名（"这个叫屋顶"）、按名称/特征引用（"那个红色的圆"）
- **P2 自由绘制**：语音控制画笔连续移动（"向右画 100，向下画 50"，turtle 风格）
- **P2 样式高级**：渐变、描边、图层顺序（"把太阳放到云后面"）
- **P2 保存/导出**：导出 PNG（语音触发，下载）

## 5. 开发计划（5 个工作日，每阶段有可验收产物）

### M0 — 工程骨架 + 绘图引擎（Day 1）
- Vite + React + TS + Konva 工程；DSL Schema 定义（zod）
- DSL 解释器 + 场景图 + undo/redo 命令栈
- **调试面板：文本框直接输入 DSL/自然语言**（整个开发期不依赖语音也能测主链路）
- ✅ 验收：文本输入 DSL JSON 能正确绘制、撤销、重做

### M1 — 语音输入链路（Day 2）
- 麦克风采集 + Silero VAD 断句 + 状态机（listening/processing/speaking）
- 接入七牛云流式 ASR（经 Node 代理），partial/final 转写上屏
- ✅ 验收：说话 → 实时字幕显示转写结果，断句正确

### M2 — 指令理解（Day 3）
- 规则快路径：模板 + 槽位抽取（图形/颜色/尺寸/方位词表），同音词纠错
- LLM 结构化解析：prompt 携带场景图摘要，输出 DSL Op 数组，JSON 校验 + 重试
- 路由器 + 指代消解（焦点对象）
- ✅ 验收：「画一个红色的圆」「把它变大」「在圆的左边画个方块」全链路语音完成

### M3 — 复杂拆解 + 语音反馈闭环（Day 4）
- Agent 拆解：创作型指令 → 多 Op 计划 → 逐条执行 + 进度播报
- TTS 反馈编排：成功短播报、歧义/破坏性操作语音确认（"确认/取消"）
- ✅ 验收：「画一个雪人」自动拆解绘制；「清空画布」需语音确认

### M4 — 延迟/成本优化 + 文档（Day 5）
- partial 转写投机解析；规则层命中率统计埋点
- 延迟测量（说完→上屏的分段耗时面板）；LLM 调用计数
- 整体联调、边界用例（噪音、口误、连续指令）
- 撰写最终设计文档：计划 vs 实际实现的指令能力对照、未完成原因
- ✅ 验收：简单指令端到端 < 500ms（不含说话时长），演示脚本跑通

### 风险与预案
| 风险 | 预案 |
|------|------|
| 七牛 ASR/TTS 接入文档不清或配额问题 | 开发期用 Web Speech API 占位，接口已抽象成 Provider 可热切换 |
| ASR 对绘图领域词识别差 | 领域词表 + 拼音纠错层；ASR 支持热词则配置热词 |
| LLM 输出不稳定 | JSON Schema 强校验 + 自动重试 + 失败播报，绝不让坏数据进执行引擎 |
| 复杂创作效果差 | P1 范围先做 4~5 个内置组合模板兜底，Agent 自由拆解作为增强 |

## 6. 成本控制策略（设计文档需对照"计划/实际"，此处为计划）

1. **规则快路径前置**：高频简单指令 0 成本本地处理，目标拦截 70%+ 的 LLM 调用
2. **VAD 本地化**：不向云端传静音音频，ASR 按有效语音计费
3. **场景图摘要而非全量**：LLM prompt 只带对象列表摘要（id/类型/颜色/包围盒），控制输入 token
4. **小模型优先**：解析任务用便宜模型，仅创作拆解用强模型
5. **TTS 兜底用浏览器合成**：非关键反馈走 speechSynthesis，0 成本
6. **会话内 prompt 前缀稳定**：系统提示词固定不变，命中服务端 prompt cache
