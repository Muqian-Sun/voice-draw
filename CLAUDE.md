**一、原创与议题**
- 作品须符合所选议题方向，且自主完成、**无抄袭**。
- 引用了第三方库/框架：**必须在 README 列明依赖**，并说明哪些是原创功能部分。未列明 = 无效。
- 复用了自己过去的代码片段：**必须在该 PR 描述中注明来源**。未注明 = 无效。

**二、持续交付（严禁"突击提交"）**
- 自议题发布之日起，**全周期保持持续的 PR 与 commit 记录，每天都要有 PR 和 commit**。
- **仅在最后一天一次性导入全部代码 = 直接无效。** 严禁在工作区堆积未提交改动——每完成一个功能就立即提交。
- **所有 commit 时间戳必须落在所选批次的开始与截止时间之内**，否则无效。
- 不 rebase、不导入旧历史。

**三、PR 规范（基于 PR 添加功能）**
- 一切变更走 `feat/*` 分支 → PR → merge，**禁止直接 push main**。
- **每个 PR 只做一件事**；尽可能小、粒度尽可能细；大功能拆成多个独立 PR。
- 标题：一句话说明本 PR 新增/修改了什么。
- 描述四要素缺一不可（模板见下《PR 描述规范》）：**功能描述 / 实现思路 / 测试方式 / 依赖与来源声明**。描述空白或与实际代码变更严重不符 = 无效。
- **每个 PR 合并后 main 必须保持可运行**：评委任意时间 clone 都能复现演示。强依赖外部服务的功能须先有降级路径再合并（WebSpeech 兜底、调试面板文本输入）。

**四、交付物**
- 公开仓库 + README + demo 视频（外链放 README 顶部，无登录可播放）；赛事截止后仓库须转为公开。

**五、执行纪律（给 Claude 的硬性要求）**
- commit / push / 开 PR / 合并属对外操作，执行前需用户授权；但上述合规要求本身是硬性的——一旦发现工作区有未提交积压，**必须主动提示并尽快按小 PR 补提交，绝不放任堆到最后**。
