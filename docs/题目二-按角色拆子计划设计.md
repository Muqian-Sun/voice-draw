# 按角色拆子计划（多主体构图）设计 RFC

> 状态：**设计 / 待实现**（2026-06-14 提出）。配套《持久锚定与几何接地设计》——子计划治"初始放置"，锚定治"事后修正"，互补。

## 1. 动机

多主体场景（「白雪公主和七个小矮人」）走**一个扁平 71-op plan** → LLM 在整张 1024×768 里放 ~70 个部件 → 坐标错（手偏、角色挤、甚至画到画布外）。根因：**LLM 在大坐标空间放多部件不可靠**；实测把几何接地放在 create 期、靠 LLM 逐件输出 `at` 也失败（GH #100→#101 回退，LLM 半依从把手画到原点）。

子计划用 LLM 的**两个强项**——① 高层语义布局（哪个角色在哪、多大）② 在聚焦上下文里画好**单个**角色——绕开它的弱项（一锅画多角色 + 大画布盲算坐标）。

## 2. 架构（三段，前端编排、复用现有 plan 管线）

1. **布局规划（planner LLM 调用，新 mode/prompt）**：输入用户话术，输出
   `{ background?: string, style?: string, subjects: [{ label, cx, cy, w, h }] }`
   ——画布上每个角色的中心 + 包围框（+ 可选背景、统一风格提示）。LLM 擅长这种语义布局。
2. **背景**（若 planner 给了）：一次普通 plan/op 铺满画布（z 最低）。
3. **逐角色子计划**：对每个 subject 调现有 plan，指令"把 {label} 画在中心 (cx,cy)、约占 w×h"——LLM 在**小区域**画该角色 ~10 件、误差有界、内部自洽（手挨着袖）。每角色部件**编为一组**（组名=label）。

## 3. 坐标策略

- **v1**：每角色 plan 直接画**绝对坐标、居中于 (cx,cy)、占 w×h**——复用现有"主体居中铺画布 60~80%"的成熟能力，只是把"画布"换成"子框"。不引局部帧+平移，最小改动。
- **后续可选**：LLM 画归一化局部帧（0,0 起）+ 编排把整组平移到 (cx,cy)——更干净（LLM 永远在同一小帧里画），但需平移整组（vpath 设 x,y、图元改位）。

## 4. 编排

- **位置**：**前端**（App.tsx 新增 orchestrator 模块），复用 `parseWithLlmStream`/`execOps`/`autoGroup`/`commitIncremental`/history。后端维持瘦代理（仅多一个 planner 转发，或复用 LLM 转发加 mode）。
- **v1 串行**：背景 → 角色1 → 角色2 …（用户看角色逐个浮现，符合"看绘制过程"）。延迟 ~N×5s（8 角色 ≈ 40s）。
- **整套作为一个事务**：N 个子计划的对象一次性 `commitIncremental`（一次快照）→ **undo 一次退整幅**，与现有 plan 语义一致。

## 5. 触发

planner 返回**多** subject → 走子计划编排；返回**单** subject → 退回现有普通 plan（不变）。检测在 planner 这一步天然完成，无需额外启发式。

## 6. 与现有机制交互

| 机制 | 处理 |
|---|---|
| 现有 plan 管线 | 每个子计划复用 parseWithLlmStream/forwardRetry/execOps，无需新引擎 |
| autoGroup | 每角色一组（组名=label）；背景不编组 |
| op 预算（#95 放宽到 50） | 每子计划是单主体、回到 8~26；50 上限留着无害 |
| 持久化/undo | 整套一次快照；undo 退整幅 |
| 半双工/TTS | 编排期间静默，全部画完再播完成语 + 建议 |

## 7. 风险 / 待定

- **planner 布局质量**：框可能重叠/比例怪/超界。缓解：planner prompt 强约束（互不重叠、留间距、主角大配角小、全落画布内、给坐标范围示例）。
- **延迟**：8 角色串行 ~40s。v1 接受（复杂场景本就慢），Phase 2 并行子调用 → ~3 轮。
- **单角色失败容错**：某子计划失败 → 跳过该角色、继续其余（不整盘崩；复用"画成任意即提交"思路）。
- **风格一致**：各角色独立画风格可能略飘。缓解：planner 输出统一 `style` 提示，透传给每个子计划。
- **成本**：N+1 次 LLM 调用（8 角色 = 9 次）。多主体才触发，单主体不变。

## 8. 分阶段

- **Phase 1（最小可测）**：planner 调用 + 串行逐角色子计划 + 编组 + 一次提交。**先验证"拆开画是否真的更好"**，再投优化。
- **Phase 2**：并行子调用、流式逐角色、planner 布局约束打磨、单角色失败容错。
- **Phase 3**：与持久锚定结合（角色内连接件用 attach）、风格一致性、子框局部帧+平移。

## 9. 与《持久锚定》RFC 的关系

互补、各治一半：

- **子计划**：保证**创建时**每个角色内部自洽（手挨着袖，因为在小帧里一起画、LLM 算的范围小）。**初始质量主杠杆。**
- **持久锚定**：保证**之后改得动 + 父件变了不脱节**。**修正杠杆。**
