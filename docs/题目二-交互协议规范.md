本文档定义系统各层之间的数据契约，是实现的唯一依据。配套文档：

协议分四部分：

3. **语音协议**（ASR / TTS Provider 抽象与代理转发协议）
4. **运行时事件与状态机**（模块间事件总线）

## 1. 绘图 DSL（Drawing Protocol）

### 1.1 设计原则

- **原子性**：每个 Op 是一次不可分的画布变更，对应 undo 栈中的一帧
- **声明式**：Op 描述"要什么"（在房子左边），坐标解析由执行引擎完成，LLM 不必算绝对像素
- **可校验**：全部类型有 zod Schema，非法 Op 在进入执行引擎前被拒绝
- **来源无关**：规则层、LLM 层、Agent 层、调试面板输出的都是同一套 Op

### 1.2 坐标系与画布

- 逻辑画布固定 **1024 × 768**，原点左上角，x 向右、y 向下。物理尺寸由前端自适应缩放，所有协议内坐标均为逻辑坐标。
- 越界策略：执行引擎对超出画布的坐标 **clamp 到边界并继续执行**（不报错，宁可画在边上也不中断语音流）。

### 1.3 核心类型定义（TypeScript / zod）

```ts
// ---------- 目标选择器：指令如何引用画布上的对象 ----------
type TargetSelector =
  | { byId: string }            // "circle#3" — 引擎生成的唯一 id
  | { byName: string }          // 用户命名："屋顶"
  | { byFocus: true }           // 焦点对象（最近创建/操作的对象），即 “它 / 刚才那个”
  | { byQuery: {                // 按特征查询："那个红色的圆"
      shape?: ShapeKind;
      fill?: Color;
      ordinal?: "first" | "last" | number;  // "第二个圆"
    }};

// 解析规则：byQuery 命中 0 个 → 执行错误 TARGET_NOT_FOUND；
// 命中多个且无 ordinal → 错误 AMBIGUOUS_TARGET（携带候选列表，触发语音澄清）

// ---------- 位置：绝对或相对 ----------
// Gradient: { from: Color; to: Color; angle?: number }（v1.6 两色线性渐变）
type Anchor = "center" | "top" | "bottom" | "left" | "right"
            | "top-left" | "top-right" | "bottom-left" | "bottom-right";

type Position =
  | { x: number; y: number }                       // 绝对逻辑坐标
  | { between: [TargetSelector, TargetSelector];   // v1.5 两参照中心插值（"脖子在头和身体之间"）
      t?: number; offset?: [number, number] }      //   t=0~1，缺省 0.5（中点）
  | { ref: "canvas" | TargetSelector;              // 相对定位："在房子左边"
      anchor: Anchor;                              //   = { ref:{byName:"房子"}, anchor:"left",
      offset?: [number, number];                   //       offset:[-外边距, 0] }
      gap?: number;                                // gap: 与 ref 包围盒的间距，外贴默认 20
      inside?: true };                             // ref 为对象时内贴（"门在房子底边"），
                                                   //   贴参照对应边内侧，gap 改为内边距（默认 0）；
                                                   //   ref=canvas 恒为内贴，此字段无效
// v1.2：line/polyline 显式给 points 且 ref 为对象时，锚定的是 points 首点（端点贴合，
// gap 默认 0）——"手臂长在身体上"类连接部件；无显式 points 的线维持包围盒语义
// v1.3：onEdge: true —— 部件"骑"在参照"真实形状边缘"的 anchor 方向交点上
// （圆按圆周/椭圆参数化/其余按 bbox 周界；沿方向 1/3 咬合 2/3 外露，线类钉首端点）。
// 视觉上"长在参照上"（耳朵/帽子/轮子）。包围盒锚定对曲线形状的斜向方位天然失效
// （圆的 bbox 角上没有圆），故增设此原语

// ---------- 尺寸：绝对或相对 ----------
type SizeSpec =
  | number                                          // 绝对值（圆半径/边长等）
  | { relativeTo: TargetSelector; factor: number }  // "比房子矮一点" → factor: 0.7
  | "small" | "medium" | "large";                   // 语义尺寸 → 引擎映射 40/80/160

type Color = string;  // CSS 颜色。中文颜色词在理解层完成映射（"天蓝色"→"#87CEEB"）

type ShapeKind = "circle" | "ellipse" | "rect" | "triangle"
               | "line" | "polyline" | "star" | "text" | "path";
```

### 1.4 操作指令集（Op）

```ts
type Op =
  // ===== 创建 =====
  | { op: "create"; shape: ShapeKind; name?: string;   // name：可选的语义名，供后续引用
      at?: Position;                                   // 缺省 → 引擎自动布局（画布空白区域）
      size?: SizeSpec;                                 // 含义随 shape：圆=半径，矩形=宽（高默认0.75宽）
      width?: SizeSpec; height?: SizeSpec;             // 显式宽高（优先于 size）
      points?: [number, number][];                     // line/polyline/path 用
      tension?: number;                                // v1.7 line/polyline/path 曲线平滑(Catmull-Rom)：0=折线(缺省) 0.4~0.5=顺滑曲线；有机轮廓(云/水波/山丘/花瓣/树冠/动物)用它
      from?: TargetSelector; to?: TargetSelector;      // v1.5 连接线（仅 line）：端点各贴 from/to 真实边缘，与 points/at 互斥
      text?: string; fontSize?: number;                // text 用
      fill?: Color; gradient?: Gradient;               // v1.6 渐变填充（优先于 fill）：{from,to,angle}
      stroke?: Color; strokeWidth?: number; opacity?: number;  // v1.6 opacity 创建即可设
      shadow?: boolean | {color?,blur?,offset?:[dx,dy],opacity?}; // v1.7 投影（立体/精致；style 亦可，false 去影）
      pattern?: 'stripes'|'dots'|'grid'|'hatch'|'cross'; // v1.7 纹理填充（叠 fill 底色，仅闭合形状）
      cornerRadius?: number;                           // v1.6 rect 圆角
      innerRadius?: number; angle?: number;            // v1.6 arc：内半径(>0圆环弧)/扇形角度(缺省270)
      rotation?: number;                               // 角度，顺时针
      desc?: string }                                  // 人类可读描述，Agent 计划播报进度用

  // ===== 修改 =====
  | { op: "style";  target: TargetSelector;            // 改外观
      fill?: Color; stroke?: Color; strokeWidth?: number; opacity?: number }
  | { op: "move";   target: TargetSelector;            // 二选一：to 或 delta
      to?: Position; delta?: [number, number] }        // "往右移一点" → delta:[60,0]（"一点"=60px）
  | { op: "resize"; target: TargetSelector;
      scale?: number;                                  // "变大一点"=1.3，"大一倍"=2
      to?: { width?: SizeSpec; height?: SizeSpec } }
  | { op: "rotate"; target: TargetSelector; degrees: number }  // 正=顺时针
  | { op: "setText"; target: TargetSelector; text: string }

  // ===== 结构 =====
  | { op: "delete";  target: TargetSelector }
  | { op: "rename";  target: TargetSelector; name: string }    // "这个叫屋顶"
  | { op: "group";   targets: TargetSelector[]; name?: string }
  | { op: "ungroup"; target: TargetSelector }
  | { op: "zorder";  target: TargetSelector;
      to: "front" | "back" | "forward" | "backward"           // 全局四档
        | { above: TargetSelector } | { below: TargetSelector } }  // v1.5 相对层级："把太阳放到云后面"={below:{byName:"云"}}

  // ===== 排布（v1.5 表达力扩展） =====
  | { op: "mirror";  target: TargetSelector; about: TargetSelector;   // 镜像复制对称部件（轴=about中心）
      axis?: "vertical" | "horizontal"; name?: string; desc?: string }  // vertical=左右镜像（缺省）
  | { op: "radial";  target: TargetSelector;                           // v2.x 放射复制：绕 about 中心等角复制 count 份（含原件）
      about: TargetSelector | { x: number; y: number };                // about=旋转中心（对象取其中心，或显式坐标）
      count: number;                                                    // 总份数（含原件），2~64；步长 360/count 度
      name?: string; desc?: string }                                    // 引擎精确均布——花瓣/光芒/轮辐/雪花臂/齿轮齿
  | { op: "align";   targets: TargetSelector[]; axis: "x" | "y" }      // 各对象中心对齐到首个目标
  | { op: "distribute"; targets: TargetSelector[]; axis: "x" | "y" }   // 首尾不动，中间等距分布

  // ===== 几何接地（v2.3） =====
  | { op: "attach";                                    // 持久锚定：建立或重解析锚定关系
      target: TargetSelector;                          // 被锚定对象（连接件：手/脚/耳/帽）
      to?: TargetSelector;                             // 父件（byName/byId）；仅 target 时 = 重解析已存 anchor
      parentAnchor?: Anchor;                           // 贴父件哪条边（如 bottom）
      mode?: "onEdge" | "outside" | "inside";         // 贴附模式（缺省 onEdge）
      childAnchor?: Anchor;                            // 子件自身哪个点去贴（缺省 center；手用 top=手腕侧）
      offset?: [number, number];
      gap?: number }

  // ===== 系统 =====
  | { op: "undo"; steps?: number }                     // "撤销三步" → steps:3，默认 1
  | { op: "redo"; steps?: number }
  | { op: "clear" }                                    // 破坏性：必须先经语音确认流程（见 §4.3）
  | { op: "focus"; target: TargetSelector }            // "选中那个圆" — 只改焦点，不改画布
  | { op: "export"; format: "png" };
```

### 1.5 执行语义与错误码

- 一次解析结果（Op 数组）作为**一个事务**进 undo 栈：用户说"画一个雪人"产生 8 个 Op，一句"撤销"整体回退。`undo/redo/export/focus` 不入栈；`clear` 作为普通事务入栈、可被撤销（规格 §5.4）。
- `undo/redo` 必须单独成事务（与其他 Op 混合视为 INVALID_OP）；规则层天然满足，LLM 被禁止输出这两类 Op。
- 执行引擎逐 Op 执行；中途失败时**保留已成功的 Op**，错误进入反馈编排器播报。

| 错误码 | 含义 | 反馈策略 |
|--------|------|---------|
| `TARGET_NOT_FOUND` | 选择器无命中 | TTS:"画布上没有找到 X" |
| `AMBIGUOUS_TARGET` | 多个候选且无法消歧 | TTS 列举候选请求澄清（"有两个圆，是红色的还是蓝色的？"） |
| `INVALID_OP` | Schema 校验失败 | 不播报技术细节，TTS:"这个操作我没理解，请换个说法" |
| `NOTHING_TO_UNDO` | undo 栈空 | TTS:"已经没有可以撤销的操作了" |
| `NOTHING_TO_REDO` | redo 栈空 | TTS:"没有可以重做的操作" |
| `UNSUPPORTED_OP` | 开发期临时：该执行能力尚未实现（按 PR 计划逐步消除，交付前应不可达） | TTS:"这个能力还没就绪" |

### 1.6 场景图（执行引擎对外暴露的只读状态）

场景图既是 LLM 的上下文输入，也是规则层指代消解的依据：

```ts
// v2.3 分层场景上下文（GH #120）
interface CanvasMapEntry {
  groupId: string;
  label: string;               // 组名
  bbox: [number, number, number, number];  // union bbox [x,y,w,h]
  memberNames: string[];       // 成员名清单
  shapeCount: number;
}

interface SceneSummary {
  canvas: { width: 1024; height: 768 };
  // Layer 1：画布地图——所有组的 union bbox + 成员名清单，永远全量，不受截断
  canvasMap: CanvasMapEntry[];
  // Layer 2：聚焦详情——focus 所在组 + utterance 提及的组/对象，部件级细节
  // 总量 ≤ DETAIL_BUDGET(40)；其余组只在 canvasMap 层
  details: Array<{
    id: string;                // "circle#1"
    name?: string;             // 用户命名
    shape: ShapeKind;
    fill?: Color; stroke?: Color;
    center: [number, number];
    bbox: [number, number, number, number];
    z: number;
    groupId?: string;
  }>;
  truncated?: boolean;         // 是否有组因预算不足未在 details 展开（地图层永远全）
  focusId?: string;            // 当前焦点对象
  lastTransaction?: { utterance: string; opCount: number };
}
```

> **v2.3 变更说明**：§1.6 场景图结构由旧版扁平 `objects[]`（超 30 个截断）升级为 `canvasMap + details` 两层，配套 `buildSceneSummary` 重构（GH #120）。`canvasMap` 永远给全部组的摘要信息，让 LLM 在任意复杂画布上都知道完整角色/组清单；`details` 只展开当前焦点相关的组的部件细节，控制 token 预算。旧版 `objects` 字段已废弃。

## 2. LLM 输入/输出协议

### 2.1 调用方式

- 单一 chat completion 接口，**强制 JSON 输出**（response_format=json + Schema 校验）
- System prompt **逐字节稳定**（DSL 精简规格 + 输出规则 + few-shot），以命中服务端 prompt cache；所有变化信息放 user message

### 2.2 输入（user message 的 JSON 载荷）

```jsonc
{
  "utterance": "在房子左边画一棵比它矮的树",        // ASR final 结果（已过同音词纠错）
  "asr_alternatives": ["在房子左边花一颗比它矮的树"], // ASR 备选（≤2 条），供 LLM 纠错参考
  "scene": { /* SceneSummary，§1.6 */ },
  "recent": [                                       // 最近 3 轮成功指令，供多轮指代
    { "utterance": "画一个棕色的房子", "summary": "create rect 房子" }
  ],
  "mode": "parse",                                  // "parse" = 单轮解析；"plan" = 创作拆解
  "image": "data:image/png;base64,..."              // 可选（v1.2）：画布渲染截图，视觉自检指令
}                                                   //   （"检查一下画面"）附带；带图时代理自动
```                                                 <!-- 切多模态模型（按需多模态，常规指令不带图） -->

`mode` 由路由器决定：规则层判断到创作型关键词（画+组合名词、"一幅"、风格词）→ `plan`，其余升级到 LLM 的走 `parse`。两种 mode 共用同一 System prompt 的不同小节，输出协议相同。

**Token 控制（v2.3 分层上下文）**：`scene` 采用 `canvasMap + details` 两层结构（GH #120）。`canvasMap` 永远给全部组的摘要（组名/union bbox/成员清单），不受截断。`details` 只对 focus 所在组 + utterance 提及名字的组展开部件级细节，总量 ≤ 40 件；超出预算的组只在地图层，并附 `"truncated": true`（含义：是否有组未在 details 展开，地图层永远全）。

### 2.3 输出（LLM 必须返回且仅返回此 JSON）

```jsonc
{
  "intent": "ops" | "clarify" | "reject",
  "confidence": 0.92,                  // 0~1，解析把握
  "ops": [ /* Op[]，intent=ops 时非空，否则必须为空数组 */ ],
  "say": "好了，房子左边画了一棵树",     // TTS 播报文案，≤20 字；intent=ops 时必填
  "clarify": {                         // 仅 intent=clarify 时存在
    "question": "有两个圆，你说的是红色的还是蓝色的？",
    "expecting": ["红色", "蓝色"]      // 下一轮语音的候选答案，供路由器快速匹配
  },
  "reason": "用户想画画以外的东西"      // 仅 intent=reject 时存在，不播报原文
}
```

**协议约束（Schema 之外的业务校验，违反即视为解析失败）：**
1. `intent=ops` ⇔ `ops.length > 0`；clarify/reject 时 ops 必须为空
2. `ops` 中不得出现 `clear`（破坏性操作只允许规则层产生，走确认流程）
3. `plan` 模式下每个 create Op 必须带 `desc`（进度播报用），总 Op 数 ≤ 28
4. `confidence < 0.6` 时路由器改走 clarify 流程，即使 intent=ops

**失败与重试策略：**

```
LLM 返回 → JSON.parse / zod 校验
  ├─ 通过 → 业务校验 → 执行
  └─ 失败 → 重试 1 次（原输入 + 追加校验错误信息）
              └─ 再失败 → TTS "没听懂，请换个说法"；本轮丢弃
超时：parse / plan 模式均 8s / 20s 无首 token 则中止并播报。
思考模式（v1.7 终）：**parse / plan 均关思考以求出图速度**。曾给 plan 开思考提升构图，但服务端思考 10~25s 才出首内容 token（"几十秒还在解析中"），体验差；改为**关思考 → 首内容 token ~1.8s 秒出 → 流式渐进绘制 → 画后多轮异步视觉自检精修**兜底质量（不追求一次成图）。首个部件上屏即把状态切到 EXECUTING（绘制中）。
```

### 2.4 System Prompt 结构（固定不变，约 1.2k token）

```
[角色] 你是语音绘图工具的指令解析器，把中文口语转为绘图 JSON 指令。
[DSL 规格] §1.3/§1.4 的精简版（仅字段名+一行说明）
[规则] 输出约束（§2.3 的 4 条）、颜色词映射表、模糊量词映射
       （"一点"=60px / "很多"=200px / factor 0.7、1.3 等）
[场景理解] 如何使用 scene 做指代消解；优先 byName > byFocus > byQuery
[plan 模式] 组合图案拆解原则：从背景到前景、从大到小、≤20 个 Op、
            部件间用相对定位（ref 前一部件）保证整体协调
[few-shot] 4 个示例：简单创建 / 相对定位 / 指代修改 / 雪人拆解（截断版）
```

### 2.5 规则快路径的输出（与 LLM 同构）

规则层输出统一为 `ParseResult`，路由器对上层屏蔽来源：

```ts
interface ParseResult {
  source: "rule" | "llm-parse" | "llm-plan";
  intent: "ops" | "clarify" | "reject" | "confirm-pending";  // confirm-pending：规则层独有，
  ops: Op[];                                                  // 破坏性操作等待语音确认
  say?: string;
  confidence: number;        // 规则命中恒为 1.0
  latencyMs: number;         // 埋点：解析耗时
}
```

## 3. 语音协议（ASR / TTS）

### 3.1 Provider 抽象（前端接口，可热切换实现）

```ts
// 会话式接口：VAD 持有唯一音频管线并负责断句，Provider 只消费帧。
// onSpeechStart → startSession()；说话期间逐帧 pushAudio()；onSpeechEnd → endSession() → 等 final。
// WebSpeech 实现自行采音与断句，pushAudio/endSession 为空操作。
interface AsrProvider {
  readonly name: string;
  startSession(): void;
  pushAudio(frame: Float32Array): void;              // 16kHz 512 样本帧
  endSession(): void;
  dispose(): void;
}
interface AsrEvents {                                 // 构造时注入
  onPartial: (text: string) => void;                 // 增量转写，驱动字幕 + 投机解析
  onFinal: (r: { text: string; confidence: number;
                 alternatives: string[] }) => void;  // 断句后的最终结果
  onError: (code: string, recoverable: boolean) => void; // recoverable=false → 上层切换兜底
}

interface TtsProvider {
  speak(text: string, opts: { interrupt?: boolean }): Promise<void>;  // interrupt: 打断当前播报
  cancel(): void;
}
// 实现：QiniuAsr / WebSpeechAsr；QiniuTts / WebSpeechTts —— 配置切换
```

**半双工约束**：TTS 播报期间暂停 ASR 上传（麦克风会录到扬声器声音造成自激）。状态机保证 speaking 与 listening 互斥；确认流程的"确认/取消"在播报结束后的专用监听窗口（5s 超时视为取消）。

### 3.2 前端 ↔ Node 代理 WebSocket 协议（ASR 转发）

代理只做两件事：隐藏云厂商密钥、协议适配。无业务状态。

```jsonc
// ↑ 客户端 → 代理
{ "type": "start", "format": "pcm16", "sampleRate": 16000, "lang": "zh" }
{ "type": "audio", "seq": 42, "data": "<base64 pcm16 20ms帧>" }
{ "type": "stop" }                       // VAD 判定句尾时发送

// ↓ 代理 → 客户端
{ "type": "ready" }                      // start 的应答，会话就绪
{ "type": "partial", "text": "在房子左边画" }
{ "type": "final",   "text": "在房子左边画一棵树", "confidence": 0.91,
  "alternatives": ["在房子左边花一棵树"] }
{ "type": "error",   "code": "UPSTREAM_DISCONNECT", "recoverable": true }
```

- 音频帧：16kHz / 16bit / 单声道 PCM，20ms 一帧（VAD 判静音的帧不发送）；消息一律 JSON 文本（音频 base64），二进制帧拒绝
- 错误码：`BAD_MESSAGE`（非 JSON/不符协议）、`NOT_STARTED`（未 start 先 audio）、`UPSTREAM_DISCONNECT` / `UPSTREAM_ERROR`（上游异常）；均 recoverable，客户端可重新 start
- 重复 `start` = 重置会话（释放旧上游并重新 ready）
- **mock 模式（无密钥）**：代理以 mock 上游应答——partial 为 final 文本的递增前缀（每 ~300ms 语音推进一字），stop 后 ~120ms 返回 final（文本经 `MOCK_ASR_FINAL` 配置，缺省"画一个圆"），保证无密钥可跑通并演示全链路
- 断线重连：客户端指数退避重连（0.5s/1s/2s，3 次失败后 TTS 播报并切换 WebSpeech 兜底）
- 句首回填：VAD 在语音真正起始**之后**才触发，若不补偿会吃掉首字/首音节。前端保留最近 ~320ms 音频帧（ring buffer ≈10 帧），`onSpeechStart` 时回填进 ASR 会话（`preSpeechPadMs=320` 对齐），保证句首完整以提升识别准确率

**TTS 转发（HTTP，分块流式，v1.7）**：`GET /api/tts?text=`（兼容 `POST /api/tts {text}`）→ `audio/mpeg`。代理对上游（火山豆包语音合成 2.0 unidirectional）**边收 JSON-lines 音频块边解码原始 mp3 字节分块下发**（chunked），前端用 `<audio src>` 渐进播放，**首块到达即开声**——播报延迟从"整段合成+下载完成"降到"首块到达"。首块前的上游错误仍以 JSON 报错（headers 未发）；未配置密钥 → 503，前端降级 speechSynthesis。

### 3.3 LLM 转发（HTTP）

`POST /api/llm/parse` → body 即 §2.2 输入；代理附加密钥转发火山方舟 LLM。缓冲模式（缺省）：代理对上游用流式以实现首 token 超时（§2.3），聚合后返回 `{content, latencyMs, firstTokenMs, model}` 供成本/延迟埋点。

**流式交付（v1.4，body 加 `stream: true`）**：代理把 LLM content 增量透传（chunked text）。前端边收边用括号深度状态机提取 ops 数组里**每个完整的 Op**，单 Op 过 zod + 本地操作禁单后立即执行上屏（**渐进绘制**：首个部件 ~3s 可见，此前需黑等全文 ~7s）。System Prompt 约定字段顺序 intent→confidence→ops→say，使路由判断先于绘制；流式期间不入 undo 栈，流结束以完整文本做权威校验：通过 → 一次性提交快照（undo 语义与缓冲一致，整体回退）；失败 → 回滚画布、退回缓冲模式（自带一次重试）。

## 4. 运行时事件与状态机

### 4.1 主状态机

```
        ┌──────────────────────────────────────────────┐
        ▼                                              │
  ┌───────────┐  VAD句尾   ┌─────────┐  ParseResult  ┌──┴──────┐
  │ LISTENING │──────────▶│ PARSING │──────────────▶│EXECUTING│
  └───────────┘            └─────────┘                └──┬──────┘
        ▲                       │ intent=clarify          │ 完成
        │                       ▼                         ▼
        │                  ┌──────────┐  TTS结束    ┌──────────┐
        └──────────────────│ SPEAKING │◀───────────│ (播报say) │
         （确认窗口为        └──────────┘             └──────────┘
          SPEAKING 的子态 AWAIT_CONFIRM，5s 超时→取消）
```

- LISTENING 期间收到 partial → 并行触发**投机解析**（仅规则层，不调 LLM）；final 与 partial 文本一致时直接采用投机结果，省去一次解析延迟
- PARSING 期间用户继续说话 → 音频缓存排队，不丢弃（支持连续指令）
- **常态监听（v1.7）**：进入应用即自动启动 VAD 常听（无"开始聆听"按钮，浏览器首次需授权麦克风），有声自动开始、停顿约 1s 自动断句（redemptionMs=1000；原 512ms 太激进，编辑长句句中停顿会被切断致转写残缺、看着像"听不懂"，放宽以等用户说完）——本地 Silero VAD 免费门控，**仅检测到人声的语音段才打开 ASR 上游并计费**（静默不上传、不计费）。主控球是状态灯：待命(聆听中)/在听/解析/播报/已静音/已休眠，点击=静音⇄恢复；授权失败→可点重试。无声 2 分钟自动休眠（降误触、长挂更安心）。
- **共创建议（v1.7 主动性）**：plan 创作完成 + 视觉自检后，再发一次轻量 LLM 主动提议一处补充（返回 say 征询 + ops 待画），语音播报 + 画面弹建议气泡。下一句说"好/可以"→采纳绘制；说"不用"或直接下达新指令→丢弃建议并按新句继续（软窗口，25s 无回应自动撤销）。采纳判定复用 §2.6 isConfirmYes。把工具变成会主动出点子的共创伙伴。

### 4.2 事件总线（模块解耦 + 埋点口径）

```ts
type AppEvent =
  | { t: "asr.partial"; text: string }
  | { t: "asr.final"; text: string; sttMs: number }          // 句尾→final 耗时
  | { t: "parse.done"; r: ParseResult }                       // 含 source/latencyMs
  | { t: "exec.done"; txId: string; ok: boolean; err?: ErrCode }
  | { t: "tts.start" | "tts.end"; text: string }
  | { t: "cost.llm"; promptTokens: number; completionTokens: number };
// 延迟看板 = asr.final.sttMs + parse.done.latencyMs + exec 耗时，按 source 分桶统计
```

### 4.3 破坏性操作确认流程（仅 clear / delete-all 类）

```
用户:"清空画布" → 规则层 → ParseResult{intent:"confirm-pending", ops:[{op:"clear"}]}
 → TTS:"确定要清空整个画布吗？" → AWAIT_CONFIRM 窗口
   ├─ 听到肯定词（确认/对/是的/清空吧）→ 执行 ops
   ├─ 听到否定词（取消/不/算了）       → 丢弃，TTS:"已取消"
   └─ 5s 超时 / 无法匹配               → 丢弃，TTS:"已取消"
```
